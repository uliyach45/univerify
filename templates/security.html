Core Standards

Our security architecture

Every component of UniVerify is designed around well-established cryptographic standards that have been peer-reviewed and battle-tested in production environments worldwide.

SHA-256 Hashing
Every document is reduced to a 256-bit cryptographic digest. Any single-bit modification produces a completely different hash, making tampering instantly detectable with mathematical certainty.
NIST FIPS 180-4
Mutual TLS 1.3
All connections are protected by TLS 1.3. Mutual authentication requires both server and client to present valid X.509 certificates, eliminating anonymous access entirely.
RFC 8446
X.509 PKI
Digital identities are established through X.509 certificates. Each signer is bound to an RSA-2048 key pair, and the certificate fingerprint is permanently embedded in every blockchain block.
RFC 5280
Blockchain Immutability
Signatures are stored in a SHA-256 hash-chained ledger. Modifying any historical record invalidates every subsequent block, making retroactive fraud computationally infeasible.
Append-Only Ledger
TOTP Email Tokens
Update authorization uses cryptographically random 32-character tokens via CSPRNG. Tokens expire after 5 minutes and are delivered over SMTP/TLS.
CSPRNG · SMTP/TLS
Real-Time Audit Logs
Every system event — authentication attempts, signings, update requests, and tamper alerts — is timestamped and persisted to a tamper-resistant audit log.
ISO 27001 Aligned
Defense in Depth

Layered security model

UniVerify implements a multi-layer defense architecture. No single point of failure can compromise document integrity.

L1
Transport Layer — TLS 1.3
All data in transit is encrypted. Man-in-the-middle attacks are prevented by certificate pinning and mutual authentication.
L2
Identity Layer — X.509 mTLS
Client certificates bind each request to a verified identity. Impersonation requires compromising the certificate authority.
L3
Integrity Layer — SHA-256
Document fingerprints are computed and verified on every access. Any byte-level modification is detected instantly.
L4
Persistence Layer — Blockchain
Signatures are permanently chained. Retroactive modification requires recomputing every subsequent block hash.
L5
Authorization Layer — Email Token
Legitimate updates require a second-factor token. Unauthorized modifications are rejected and logged automatically.
